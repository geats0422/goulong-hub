# 测试驱动开发

在任何实现代码编写之前，你必须先写测试并看它失败。

## 铁律

```
RED → GREEN → REFACTOR
```

**禁止**: 在测试之前写任何实现代码。
**禁止**: 为了通过测试而修改测试（除非测试本身有错误）。
**禁止**: 跳过 REFACTOR 步骤。

## TDD 循环

### 第 1 步: RED — 写失败的测试
- 写一个最小的、精确的测试来验证你即将实现的行为
- 运行测试，**确认它失败了**
- 如果测试没有失败，说明要么测试写得不对，要么功能已经存在

### 第 2 步: GREEN — 写最小实现
- 写刚好足够的代码让测试通过
- 不要写超出测试覆盖范围的功能
- 不要"预见未来需求"——YAGNI (You Aren't Gonna Need It)
- 运行测试，**确认它通过了**

### 第 3 步: REFACTOR — 改进设计
- 消除重复代码
- 改善命名
- 提取公共逻辑
- 运行测试，**确认它们仍然全部通过**
- 确保覆盖率达到 80%+

## 如何写好的测试

**一个好的测试**:
- 测试一个具体的行为
- 有清晰的 Given-When-Then 结构
- 独立于其他测试
- 快速运行

**一个不好的测试**:
- 测试实现细节而非行为
- 依赖特定的数据结构
- 依赖于外部服务（除非是集成测试）
- 测试多个不相关的东西

## 测试类型

1. **单元测试** — 单个函数、工具函数、组件
2. **集成测试** — API 端点、数据库操作
3. **E2E 测试** — 关键用户流程

## 调试失败

如果测试失败：
1. 检查测试是否独立（不依赖其他测试的状态）
2. 检查 mock/stub 是否正确
3. 如果实现有 bug → 修复实现
4. 如果测试有 bug → 修复测试（但这很少见——不要首先假设测试有问题）

## 常见问题

**Q: 所有代码都需要 TDD 吗？**
A: 是的。没有例外。

**Q: 如果项目没有测试框架怎么办？**
A: 先配置测试框架。这是第一步。

**Q: 如果测试写起来很困难怎么办？**
A: 测试越难写，意味着代码设计可能有问题。考虑重构接口使测试更容易。
