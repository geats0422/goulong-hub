# Grill Me — 设计压力测试

在 brainstorming 产出设计文档后、进入 /plan 拆分任务之前，对设计方案进行压力测试。

## 在工作流中的位置

```
/brainstorm → 设计文档 → 用户批准
     ↓
★ grill-me → AI 质问用户，逐个决策分支验证，确保需求对齐 ★
     ↓
/plan → 拆解为实施任务
     ↓
/execute → 逐个执行
     ↓
/finish → 验证、提交、交付
```

## 核心规则

1. **一次只问一个问题** — 不要一次抛出多个问题
2. **每个问题给出推荐答案** — 不要只问不答
3. **能通过探索代码库回答的问题，自己探索** — 不浪费用户时间
4. **不放过模糊地带** — 发现设计文档中有歧义的地方，必须追问到清晰
5. **追问到决策树的叶子节点** — 不要停在中间层

## 提问方向

按以下维度逐一质问（每个维度 1-3 个问题）：

| 维度 | 典型问题 |
|------|---------|
| **用户场景** | "谁会用这个功能？主要场景是什么？" |
| **边界条件** | "如果 X 为空/Y 超时/Z 并发会怎样？" |
| **数据模型** | "这个字段可以为 null 吗？删除时级联吗？" |
| **错误处理** | "这个 API 调用失败后用户看到什么？" |
| **安全风险** | "这个输入需要校验吗？权限怎么控制？" |
| **性能预期** | "预期的并发量？响应时间要求？" |
| **兼容性** | "改了这个接口，旧客户端会怎样？" |
| **可测试性** | "这个行为怎么验证？有 observable 的输出吗？" |

## 输出格式

每个问题：

```
**问题**: [具体问题]
**推荐答案**: [AI 的建议]
**你的选择**: [等待用户回答]
```

## 终端状态

当所有决策分支都清晰后：
1. 输出"需求对齐总结"（3-5 条关键决策）
2. 如果设计文档有遗漏，补充到文档中
3. 告诉用户可以运行 `/plan` 开始拆分任务
