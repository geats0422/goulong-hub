# 先搜索再构建

在编写代码之前，停下来先搜索是否有现成的解决方案。

## 三层知识体系

### 第一层: 经典验证
标准的、经过实战检验的模式和方法。你很可能已经知道这些。但仍值得快速确认——偶尔质疑经典方案是产生创新的地方。

### 第二层: 新兴流行
当前最佳实践、博客文章、生态趋势。搜索这些，但批判性地审视。人群对新事物的判断可能和旧事物一样错误。

### 第三层: 第一性原理
从对具体问题本身的推理中得出的原创观察。这是最有价值的知识。珍视它胜过一切。

## 搜索步骤

收到任何涉及不熟悉技术/模式/基础设施的请求时：

1. **搜索 "{技术} {需求} 内置方案"**
   - 运行时/框架可能已经有内置的功能
   - 例如: "Node.js rate limit built-in" → 可能有中间件

2. **搜索 "{需求} 最佳实践 {年份}"**
   - 避免使用过时的方法
   - 例如: "error handling best practice 2026"

3. **检查官方文档**
   - 官方文档 > 博客 > Stack Overflow

## 反模式

- ❌ 从零实现一个运行时/框架已经内置的功能（第一层错误）
- ❌ 不加批判地接受博客文章中的建议（第二层狂热）
- ❌ 假设"经典方案"总是对的而不质疑前提（第三层失明）

## 搜索后的决策

搜索完成后，你的输出应包含：
```markdown
## 搜索结果
- 现有方案: [找到的库/工具/模式]
- 为什么选择/不选择: [理由]
- 如果选择现有方案: 如何集成
- 如果没有合适方案: 为什么需要从零构建
```

## 注意事项

- 搜索成本趋近于零——永远先搜索
- 最坏的情况是花一天构建一个已有一行代码能解决的问题
- 最好的情况是构建一个没人想过的东西——因为你搜了，理解了全貌，看到了别人没看到的
