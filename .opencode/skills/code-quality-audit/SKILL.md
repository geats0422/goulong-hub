# 代码质量审计

自动识别项目类型，使用对应的评估标准进行全面的代码质量审计。

## 触发条件

- 用户运行 `/quality` 命令
- 用户说"评估代码质量"、"代码审计"、"质量怎么样"
- 用户说"评估前端/后端/数据库的代码质量"

## 第一步：自动检测项目类型

运行以下检测（按顺序），确定评估框架：

```
检测 1: 前端项目?
  存在 tsconfig.json 或 jsconfig.json
  + 存在 React/Vue/Angular/Svelte 依赖
  + 或存在 src/App.tsx / src/main.ts 等入口
  → 是 → 使用"前端评估框架"

检测 2: 后端项目?
  存在 package.json 且有 express/fastify/koa/nest/gin/echo/django/flask 等依赖
  + 或存在 go.mod + 路由代码
  + 或存在 requirements.txt + 路由代码
  → 是 → 使用"后端评估框架"

检测 3: 数据库密集?
  存在 migrations/ 或 prisma/ 或 knex 配置
  + 或存在大量 *.sql 文件
  + 或 connection pooling 配置
  → 是 → 加入"数据库评估维度"

检测 4: 全栈?
  前端 + 后端检测都通过
  → 是 → 使用"全栈评估框架"（合并前后端维度）

检测 5: 通用项目
  以上都不匹配
  → 使用"通用评估框架"
```

## 评估框架

### A. 前端评估框架

| 维度 | 权重 | 检查内容 |
|------|------|---------|
| 类型安全 | 20% | TypeScript 严格模式、`any` 使用 |
| 组件设计 | 20% | 组件大小、职责单一、Props、错误边界 |
| Lint 规范 | 15% | ESLint/Biome 规则遵守 |
| 性能 | 15% | 重渲染、memo、代码分割、懒加载 |
| 测试覆盖 | 10% | 覆盖率、关键路径测试 |
| 可访问性 | 10% | ARIA、语义化、键盘、对比度 |
| 代码组织 | 10% | 文件结构、循环依赖、关注点分离 |

### B. 后端评估框架

| 维度 | 权重 | 检查内容 |
|------|------|---------|
| API 设计 | 20% | REST 规范、状态码、错误格式、分页、版本控制 |
| 安全性 | 20% | 认证、授权、输入验证、SQL 注入、密钥管理 |
| 错误处理 | 15% | 一致的模式、日志、不泄露实现细节 |
| 数据库访问 | 15% | N+1 查询、连接池、迁移、参数化查询 |
| 中间件/拦截器 | 10% | 职责清晰、顺序正确、无冗余 |
| 测试覆盖 | 10% | 集成测试、API 测试 |
| 代码组织 | 10% | 分层架构（controller/service/repository）、模块边界 |

### C. 数据库评估框架

| 维度 | 权重 | 检查内容 |
|------|------|---------|
| Schema 设计 | 25% | 范式化、约束（FK/PK/unique）、字段类型合理 |
| 索引策略 | 20% | 缺失索引、冗余索引、复合索引顺序 |
| 查询质量 | 20% | 参数化、N+1 避免、复杂查询的 EXPLAIN 分析 |
| 迁移管理 | 15% | 可回滚、无破坏性变更、版本有序 |
| 连接管理 | 10% | 连接池配置、超时设置、泄漏检测 |
| 数据安全 | 10% | 敏感数据加密、访问控制、审计日志 |

### D. 全栈评估框架

合并前端(A)和后端(B)的维度，调整权重：

| 维度 | 权重 |
|------|------|
| 前端: 类型安全 + 组件设计 | 各 12% |
| 后端: API 设计 + 安全性 | 各 12% |
| 前端: Lint + 性能 | 各 8% |
| 后端: 错误处理 + 数据库访问 | 各 8% |
| 前端: 测试 + 可访问性 + 组织 | 各 6% |
| 后端: 测试 + 组织 | 各 6% |
| 数据库维度（如有） | +5%（从其他维度均分） |

### E. 通用评估框架

| 维度 | 权重 | 检查内容 |
|------|------|---------|
| 代码组织 | 25% | 模块结构、命名约定、关注点分离 |
| 错误处理 | 20% | 一致的模式、不静默吞错误 |
| 测试覆盖 | 20% | 测试存在性、覆盖率 |
| 代码风格 | 15% | 一致性、可读性、行长度 |
| 安全性 | 10% | 密钥管理、输入验证 |
| 文档 | 10% | README、代码注释、API 文档 |

## 审计工作流

### 阶段 1: 收集定量数据 (~2 min)

按项目类型运行对应的质量工具：

**前端项目：**
```bash
npx tsc --noEmit --pretty false 2>&1 || true    # 类型检查
npx eslint . --format json 2>&1 || true         # Lint
npm test -- --coverage 2>&1 || true             # 测试
```

**后端项目（Node）：**
```bash
npx tsc --noEmit --pretty false 2>&1 || true    # 类型检查
npx eslint . --format json 2>&1 || true         # Lint
npm test -- --coverage 2>&1 || true             # 测试
# 安全: npx audit 或 npm audit
```

**后端项目（Python）：**
```bash
python -m pylint src/ 2>&1 || true              # 代码质量
python -m mypy src/ 2>&1 || true                # 类型检查
python -m pytest --cov 2>&1 || true             # 测试
python -m bandit -r src/ 2>&1 || true           # 安全
```

**后端项目（Go）：**
```bash
go vet ./... 2>&1 || true                       # 静态分析
go test -cover ./... 2>&1 || true               # 测试覆盖
```

**数据库项目：**
```bash
# 检查 migration 文件的数量和状态
# 查找 N+1 查询模式: grep -r "for.*find\|forEach.*query" src/
# 检查是否有 EXPLAIN / 查询计划
```

### 阶段 2: 定性分析 (~3 min)

使用 Read 工具逐文件审查对应框架的检查清单。

**通用检查（所有类型）：**
- 文件是否过大（> 800 行）？
- 是否存在空的 catch 块？
- 是否有硬编码密钥？
- 是否有未使用的导入和死代码？
- 命名是否清晰一致？

### 阶段 4: 优化执行（如果用户批准）

如果总分 < 60，自动建议优化：

```
优化前: 42/100
  ↓ /execute (按优先级逐任务修复)
优化后: 78/100 (+36)
```

优化流程:
1. 所有 CRITICAL 问题 → 自动创建修复任务
2. HIGH 问题 → 建议修复，征得同意后执行
3. 每次修复后 commit
4. 全部修复后重新运行审计，对比前后评分

## 评分标准

### 类型安全 (通用，0-10)
- 10: TypeScript strict / Python mypy strict / Go（语言原生强类型）
- 7: 配置了类型检查但非最严格模式
- 4: 有类型注解但大量 `any` / 未检查
- 0: 无类型系统

### API 设计 (后端，0-10)
- 10: RESTful 规范，一致的状态码和错误格式，分页，版本控制
- 7: 基本规范，少数不一致
- 4: 混合风格，缺乏一致性
- 0: 无 API 设计规范

### 安全性 (后端，0-10)
- 10: 输入验证、参数化查询、密钥环境变量、限流、HTTPS
- 7: 基本安全措施就位，少数遗漏
- 4: 存在已知风险（密钥硬编码等）
- 0: 无安全意识

### Schema 设计 (数据库，0-10)
- 10: 规范化设计，完整约束，字段类型精确
- 7: 基本规范，少数冗余或缺失约束
- 4: 明显设计问题（缺少外键、EAV 反模式等）
- 0: 无 schema 管理

### 通用评分映射
- 0 个 lint 错误 → 10
- < 5 个 lint 错误 → 8
- 5-20 个 → 6
- 20-50 个 → 4
- > 50 个 → 2
- 无工具 → 0

## 下一步

- 总分 ≥ 80: 告知用户"质量良好"，列出剩余改进建议
- 总分 60-79: 告知用户"质量尚可"，给出主要改进方向
- 总分 < 60: 建议运行优化，列出 Top 3 必须修复的问题
- 报告已保存: 告知路径
