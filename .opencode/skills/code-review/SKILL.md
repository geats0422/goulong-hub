# 代码审查

在完成代码修改后，对代码进行全面审查。

## 审查清单

### 结构完整性
- 代码是否正确实现了功能需求？
- 设计文档中的所有要点都已覆盖？
- 是否有遗漏的边界情况？

### 安全性
- 是否有硬编码的密钥/密码/token？
- 用户输入是否经过验证和清理？
- SQL 查询是否使用参数化查询？
- 是否存在 XSS/CSRF 风险？

### 代码质量
- 函数是否小巧聚焦（< 50 行）？
- 命名是否清晰合理？
- 是否有明显的重复代码？
- 错误处理是否完善？
- 是否有空的 catch 块？

### 测试
- 是否有对应的测试？
- 测试是否覆盖了关键路径？
- 测试是否独立且快速？

### 可维护性
- 代码是否易于理解？
- 模块边界是否清晰？
- 是否有不必要的复杂性？

## 问题严重程度

| 级别 | 含义 | 处理方式 |
|------|------|---------|
| **CRITICAL** | 安全漏洞、数据丢失风险 | 必须立即修复，block merge |
| **HIGH** | 可能导致生产问题 | 应该在 merge 前修复 |
| **MEDIUM** | 代码质量问题 | 建议修复，记录追踪 |
| **LOW** | 风格建议 | 可以接受，后续改进 |

## 审查流程

1. 阅读变更的整体目的
2. 逐文件逐部分审查
3. 自动修复明显的风格/安全问题
4. 标记需要用户决策的高影响问题
5. 生成审查报告

## 输出格式

```markdown
## 审查报告

### 总体评价
[2-3 句]

### 发现的问题
- **[CRITICAL]** 文件:行号 - 问题描述 + 建议修复
- **[HIGH]** 文件:行号 - 问题描述
- **[MEDIUM]** 文件:行号 - 建议

### 自动修复
- [列出自动修复的内容]

### 通过项目
- [列出审查通过没有问题的方面]
```
