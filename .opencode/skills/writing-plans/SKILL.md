# 编写实施计划

将批准的设计文档转化为精确的、可执行的分步实施计划。

## 何时使用

- 设计文档已获用户批准
- **grill-me 已完成**（或用户明确跳过）— 确保需求已对齐
- 实现需要多个步骤（> 1 个文件修改）

如果只有一个简单步骤（单文件小改动），可以跳过本技能直接实现。

完整工作流：`/brainstorm` → **grill me** → `/plan` → `/execute` → `/finish`

## 计划格式

计划保存到 `Project/{项目名称}/docs/plans/YYYY-MM-DD-<topic>-plan.md`：

```markdown
# [功能名称] 实施计划

## 总览
[2-3 句概述实现策略和关键决策]

## 前置准备
- [ ] 检查设计文档是否已批准
- [ ] 确认开发环境就绪
- [ ] 运行现有测试确认基线通过

## 任务列表

### 任务 1: [任务名] (~3 min)
- **描述**: [具体要做什么]
- **文件**: 
  - 创建 `path/to/new/file.ts`
  - 修改 `path/to/existing/file.ts`
- **测试**: [需要写什么测试]
- **验证**: [如何确认任务完成]
- **依赖**: 无

### 任务 2: [任务名] (~4 min)
- **描述**: ...
- **文件**: 修改 `path/to/file.ts`
- **测试**: ...
- **验证**: ...
- **依赖**: 任务 1

### 任务 N: [任务名] (~5 min)
- **描述**: [最终任务]
- **验证**: [最终验收标准]
- **依赖**: 任务 2

## 并行机会
- 任务 3 和任务 4 可以并行执行（无依赖关系）

## 风险 & 缓解
| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 数据库迁移回滚失败 | 低 | 高 | 先做 dry-run |

## 测试策略
| 层级 | 内容 | 覆盖目标 |
|------|------|----------|
| 单元测试 | 核心逻辑 | 100% |
| 集成测试 | API + DB | 关键路径 |
| E2E | 用户流程 | 主流程 |
```

## 好的任务 vs 不好的任务

**好的任务**:
- "创建 User 模型和数据库迁移文件" — 清晰、独立
- "实现 JWT token 生成和验证函数" — 可独立测试
- "编写 POST /auth/login 端点" — 明确的完成标准

**不好的任务**:
- "实现认证系统" — 太大，需要拆分
- "添加所有 API 端点" — 模糊，没有具体文件
- "重构代码" — 没有明确的完成标准

## 任务大小原则

- 每个任务 **2-5 分钟** 可完成
- 一个任务通常涉及 1-3 个文件
- 如果预估超过 5 分钟，拆分为两个任务
- 每个任务必须能独立验证"完成了"
