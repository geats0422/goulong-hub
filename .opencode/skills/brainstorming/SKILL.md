# 头脑风暴：将想法转化为设计

将模糊的创意通过协作对话变成完整的设计方案。

<HARD-GATE>
在完成设计呈现并获得用户批准之前，禁止 invoke 任何实现技能，禁止编写任何代码，禁止搭建任何项目。
这适用于**所有项目**，无论表面上看起来有多简单。
违反此规则的结果：未经审视的假设导致浪费的返工。
</HARD-GATE>

## 工作流程

```
了解项目上下文
  → 如果是多子系统项目 → 标记并帮用户拆分子项目
  → 逐一提问（一次一个问题）
  → 提出 2-3 个方案并给出推荐
  → 分段呈现设计（每段后确认）
  → 用户批准 → 保存设计文档
  → 设计文档自检
  → 终端状态: 告诉用户运行 /plan
```

## 详细步骤

### 1. 了解项目上下文
- 检查已有文件结构、文档、最近提交
- 了解当前技术栈和已有的模式
- 确认要修改的范围与现有代码的关系

### 2. 逐一提问
- **一次只问一个问题** — 不要一次抛出多个问题
- 优先用选择题（降低用户决策负担），开放性问题也可以
- 追问真相，不是用户说的功能：用户说"做个日历简报"实际可能是"做个 AI 个人助理"
- 聚焦于：目标、约束、成功标准、用户场景
- 识别需求中的"隐性假设"并明确指出来

### 3. 提出 2-3 个方案
- 每个方案说明核心思路、优点、缺点
- 推荐一个并解释为什么（不要只说"方案B更好"，要解释权衡）
- 方案可以覆盖：技术选型、架构风格、实现深度

### 4. 分段呈现设计
- 按部分呈现，每部分后确认方向是否正确
- 简单部分几句话即可，复杂部分可以多展开
- 覆盖：架构、组件、数据流、错误处理、测试策略
- **设计要面向隔离和清晰性**：每个单元有明确目的、通过接口通信、可独立测试

### 5. 保存设计文档
路径: `Project/{项目名称}/docs/designs/YYYY-MM-DD-<topic>-design.md`

```markdown
# [功能名称] 设计文档

## 目标
[1-2 句话说明要达成什么]

## 用户场景
[用户会如何使用？主要流程是什么？]

## 技术方案
[选什么技术/架构？为什么？]

## 数据模型
[涉及的数据结构、表、API 契约]

## 接口设计
[关键 API / 组件接口]

## 错误处理
[可能的失败模式和应对策略]

## 测试策略
[如何验证实现的正确性？]
```

### 6. 设计文档自检
保存后快速检查：是否有占位符未填？是否有矛盾？是否有歧义？范围是否合理？

### 7. 用户审查
明确告诉用户文档路径并请用户确认。在得到明确批准之前不要继续。

## 反模式

| 反模式 | 正确做法 |
|--------|---------|
| "这太简单了，不需要设计" | 简单项目最容易被不经审视的假设浪费工作 |
| 一次问多个问题 | 一次一个，降低决策负担 |
| 只描述用户的 feature request | 追问背后的真问题 |
| 不推荐方案，让用户自己选 | 给出推荐并解释原因 |
| 一次性呈现全部设计 | 分段呈现，每段确认 |

## 下一步 — 自动判断工作流路径

设计文档保存后，**必须先评估任务复杂度**，然后告诉用户走哪条路径：

### 评估维度

| 维度 | 简单 (A) | 复杂 (B) |
|------|---------|---------|
| 需求边界 | 清晰、无歧义 | 模糊、多决策分支 |
| 技术栈 | 熟悉、已有模式 | 不熟悉、需要选型 |
| 改动范围 | 单一模块/文件 | 跨模块/跨系统 |
| 隐藏假设 | 无或少 | 多，需要逐一确认 |
| 方案确定性 | 你对方案有信心 | 你对方案不完全确定 |

### 判断规则

- **全部为 A** → 路径 1：直接 `/plan`
- **任一为 B** → 路径 2：先 "grill me" 再 `/plan`

### 输出格式

```
## 复杂度评估

| 维度 | 判定 | 理由 |
|------|------|------|
| 需求边界 | A/B | ... |
| 技术栈 | A/B | ... |
| 改动范围 | A/B | ... |
| 隐藏假设 | A/B | ... |
| 方案确定性 | A/B | ... |

**判定**: 路径 1/2
**建议**: 直接运行 /plan / 或先说 "grill me" 进行需求校准
```

用户可以覆盖你的判断（比如你判路径 1 但用户想 grill，或反过来）。

完整工作流：
- **路径 1（简单）**: `/brainstorm` → `/plan` → `/execute` → `/finish`
- **路径 2（复杂）**: `/brainstorm` → `grill me` → `/plan` → `/execute` → `/finish`

不要自动进入 implementation 或 `/plan`。
