# 开发规范

## 核心原则

### 1. 清晰优先
- 代码首先是给人读的，其次是给机器执行的
- 变量/函数命名要有意义
- 复杂的逻辑要加注释

### 2. 小即是美
- 函数要小（建议 < 50 行）
- 文件要聚焦（建议 < 400 行）
- 一个文件只做一件事

### 3. 错误处理
- 任何可能失败的地方都要处理错误
- 用户友好的错误信息
- 关键操作要有日志

### 4. 安全意识
- 不硬编码密钥/密码
- 验证所有用户输入
- 防止 SQL 注入、XSS 等常见漏洞

## 代码组织

### 文件结构
```
src/
├── features/          # 按功能模块组织
│   └── feature-name/
│       ├── index.ts   # 导出入口
│       ├── types.ts   # 类型定义
│       ├── logic.ts   # 业务逻辑
│       └── ui.ts      # UI 组件（如果有）
├── shared/           # 共享代码
│   ├── utils/        # 工具函数
│   └── constants/    # 常量
└── index.ts          # 主入口
```

### 命名规范
- 文件名: kebab-case (my-component.ts)
- 类名: PascalCase (MyClass)
- 函数/变量: camelCase (myFunction)
- 常量: UPPER_SNAKE_CASE

## Git 提交规范

### 提交信息格式
```
类型: 简短描述

详细描述（可选）

相关 Issue: #123
```

### 类型
- feat: 新功能
- fix: Bug 修复
- refactor: 重构
- docs: 文档
- test: 测试
- chore: 维护

## 测试建议

### 测试金字塔
- 单元测试: 多（70%）
- 集成测试: 中（20%）
- E2E 测试: 少（10%）

### 测试原则
- 测试行为，不测试实现
- 测试边界情况
- 保持测试快速
