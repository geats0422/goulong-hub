# 通用编码规范

适用于所有编程语言和项目的基础规范。

## 基本原则

- 保持代码简洁、可读、可维护
- 遵循目标语言的社区标准编码风格
- 优先使用不可变数据模式
- 编写自解释的代码，仅在必要时添加注释

## 禁止事项

- 不得在输出中包含敏感数据（密钥、token、密码、绝对路径）
- 不得提交未经测试的变更
- 不得绕过安全检查或验证钩子
- 不得无理由地重复已有功能
- 不得在未检查测试套件的情况下提交代码

## 文件组织

- 多个小文件优于一个大文件
- 典型文件 200-400 行，最大 800 行
- 按功能/领域组织，不要按类型组织
- 高内聚，低耦合

## 错误处理

- 在每个层级处理错误
- UI 代码提供用户友好的错误消息
- 服务端记录详细的上下文信息
- 绝不静默吞掉错误

## 输入验证

- 在系统边界验证所有用户输入
- 使用基于 schema 的验证
- 快速失败，提供清晰的错误消息
- 绝不信任外部数据

## 代码质量检查清单

- 函数小（< 50 行），文件专注（< 800 行）
- 嵌套不超过 4 层
- 正确进行错误处理，不忽略返回值
- 不存在硬编码的值
- 标识符可读且命名合理

## 提交风格

- 使用约定式提交：`feat:`, `fix:`, `refactor:`, `docs:`, `test:`, `chore:`, `perf:`, `ci:`
- 保持变更模块化
- 在 PR 摘要中解释面向用户的影响

## 持续学习

- 当用户请求分析会话、提取经验教训或运行 `/learn` 时，必须使用 `continuous-learning` 技能。
- 先读取 `~/.opencode/learnings/<project>/entries/` 中已有学习条目，再提取本次会话中的可复用模式。
- 与已有条目高度相似时优先确认并提升 confidence，不要创建重复条目；新模式才创建新学习条目。
- confidence 达到 0.7 及以上时，向用户建议升级为 rule / skill / agent prompt / command，并在用户批准后执行。

## 摩擦预防（基于 /insights 2026-06-27）

### 复杂关系型变更
- 涉及多对多关系、状态机、降级逻辑的变更，**必须走 grill-me**（不可跳过）
- 在设计文档中显式列出"边界情况矩阵"，逐一确认

### UI 变更
- 样式修改与逻辑修改**分提交**，便于回滚和 review
- 同一 UI 文件单次任务内修改不超过 3 轮，超过则回到设计阶段重新对齐

### 技术选型迁移
- 架构迁移（如 HTML→Vue）必须有明确的迁移边界和回滚策略
- 迁移过程中出现 revert 时，必须记录原因到 friction-log
