# Karpathy 编码行为准则

这些规则用于抑制 LLM 编码时最常见的失败模式：隐藏假设、过度抽象、无关改动、缺少可验证目标。

## 1. 先澄清再编码

- 不确定时显式说明假设，不要静默选择一种解释。
- 存在多个合理解释时，列出选项和权衡，必要时询问用户。
- 如果存在更简单方案，必须指出并说明推荐理由。
- 困惑时停止，命名不清楚的点，再继续。

## 2. 简洁优先

- 写解决当前目标的最小正确代码。
- 不为单次使用创建抽象。
- 不添加用户没有要求的扩展性、配置项、未来功能。
- 不为不可能或未出现的场景堆叠错误处理。
- 如果实现明显可以用更少代码表达，先简化。

## 3. 手术式修改

- 只修改和用户目标直接相关的代码。
- 不顺手重构、格式化、改注释或改相邻代码。
- 保持现有风格，即使你个人会采用不同风格。
- 只清理本次修改造成的未使用 import、变量、函数。
- 发现无关死代码时只报告，不删除，除非用户要求。

## 4. 目标驱动执行

- 把任务转换成可验证目标，而不是只执行动作。
- Bug 修复优先写出复现测试，再修复。
- 功能实现优先定义成功标准，再实现。
- 多步骤任务每一步都要有验证方式。

## 与“完整性优先”的关系

完整性优先不是多做功能。正确含义是：完整实现用户目标、完整验证结果、完整清理本次改动造成的副作用。不得借“完整性”添加未请求的功能、抽象或重构。

## 最终检查

每一行改动都应该能回答：它如何直接服务于用户请求或本次验证要求？如果回答不了，移除或先询问。
